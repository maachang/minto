## 背景・目的

既存の`modules/s3table/s3MasterTable.js`は、テーブル全体を1つのJSONファイルとしてS3に読み書きする設計になっている。この方式は実装がシンプルな反面、複数リクエストが同時に同じテーブルを更新しようとすると、テーブル全体のread-modify-writeが競合しやすい。

そこで、**1行＝1ファイル**でS3に保存する方式にすることで、異なる行同士の同時書き込みがほぼ競合しないデータベースを新たに設計する。

## 既存 modules/s3table/s3MasterTable.js との違い（使い分け）

`s3MasterTable.js`（[docs/s3MasterTable.md](https://github.com/maachang/minto/blob/main/docs/s3MasterTable.md)）と`s3IndexTable.js`（本設計）は、どちらもS3をバックエンドにしたデータベースだが、**書き込み頻度**を軸に使い分ける。

| 項目 | s3MasterTable.js（既存） | s3IndexTable.js（本設計） |
|---|---|---|
| 向いている用途 | **書き込み頻度が少なく、読み込み頻度が多い**もの | **書き込み頻度が多い**もの |
| データ格納単位 | テーブル全体で1JSON | 1行＝1ファイル |
| 書き込み競合 | テーブル全体をロック的に読み書き＝競合しやすい | 行単位のファイルなのでほぼ競合しない |
| 検索方法 | JS関数(`where`)でテーブル全件走査 | 事前定義したインデックス経由のみ |
| 汎用性 | RDBMSライクに自由な条件検索が可能 | RDBMSほど汎用的ではない（インデックス設計が必須） |

## 想定スケール

このデータベースは、あくまで**小規模（1テーブルあたり1万件程度まで）を前提としたRDBMS的な使い勝手のツール**として設計する。「1行1ファイル」により、既存の`s3MasterTable.js`（テーブル全体1JSON）と比べてinsert/updateが扱いやすくなる、という程度の利便性を主目的とする。

大量データを扱いたい場合は素直にRDS等を使うべきであり、本設計はその代替を目指すものではない。そのため、以下は許容される既知のトレードオフとする。

- `dropTable`が行数分のDeleteObjectを要すること
- 既存テーブルへの`createIndex`が全行分のバックフィルを要すること
- `update`が対象行の全インデックスエントリを作り直すコストを伴うこと
- `groupBy`/集計関数がS3側でプッシュダウンされず、絞り込み後の結果セットに対してアプリ側(JS)で計算されること（数千〜1万件程度の規模でのみ実用的）

## ディレクトリ構成

~~~
[S3Bucket名]
  +-- table.json                   … 全テーブル分のテーブル定義(カラム・インデックス定義)
  |                                    をテーブル名キーで集約した1つのファイル
  +-- table/
  |     +-- {table名}/
  |             +-- {行ファイル名}   … 1行分のデータ(JSON)
  +-- index/
        +-- {table名}/
                +-- {key1}!{key2}!.../     … インデックス定義プレフィックス(キー名の並び)
                        +-- {value1}!{value2}!.../{行ファイル名}   … 0バイトファイル
~~~

- `table.json`: 全テーブルのカラム定義・インデックス定義を、テーブル名をキーにしたオブジェクトとしてまとめて持つ設定ファイル(show tables的な一覧参照・キャッシュ効率のため、テーブル単位ファイルではなく集約ファイルとして持つ)
- `table/{table名}/{行ファイル名}`: 1行分のデータをJSONとして格納
- `index/{table名}/{key1}!{key2}!.../{value1}!{value2}!.../{行ファイル名}`: インデックスは0バイトファイルとして持ち、HEAD/LISTのみで存在確認・検索ができる（本文転送が発生しない）

## 行ファイルの命名規則

~~~
{UnixTime}_{ナノ秒}_{LambdaID}_{乱数}
~~~

- ミリ秒＋ナノ秒＋Lambda実行ID＋乱数を組み合わせることで、同時実行環境でも衝突しないユニークな名前になる
- 先頭がUnixTimeなので、時系列順にソートされたファイル一覧になる（`table/{table名}/`配下を一覧すれば時系列順の走査が可能）
- この行ファイル名は**内部的な実装詳細**であり、外部（アプリケーションコードや他テーブルからの参照）が直接この名前を保持して参照することは想定しない。行への安定した参照が必要な場合は、別途「主キー用インデックス」を用意し、常にインデックス経由で解決する

## CRUD操作の方針

- **INSERT**: 行ファイルを`table/{table名}/`配下に作成し、定義済みの各インデックスに対応するインデックスエントリ(0バイトファイル)を`index/{table名}/...`配下に作成する
- **DELETE**: 行ファイルを即座に`DeleteObject`する。インデックスエントリはこの時点では削除しない（後述の「読み取り時の自己修復」で遅延削除する）
- **UPDATE**: DELETE（行ファイル削除）＋新規行ファイルの作成（新しい一意な行ファイル名になる）として扱う。旧インデックスエントリは古い行ファイルを指したまま残るが、これもDELETE同様に自己修復で処理される

削除・更新のたびに関連する全インデックスエントリを同期的に削除する設計（tombstone方式やVacuum方式）も検討したが、以下の理由でシンプルな「即時削除＋読み取り時の自己修復」を採用することにした。

- S3の複数オブジェクトにまたがる操作は、どちらの方式でも本質的にアトミックにはできない
- 削除・更新をリクエスト処理の中で同期的に完結させる（インデックスNヶ所の削除まで含めて）と、mintoが目指す低レイテンシという目的に反する
- tombstone＋定期Vacuumは、Vacuumの実行タイミング管理や「どのインデックスが対象か」の特定ロジックが余分に必要になり複雑
- 「インデックス経由でGETしたら404」を検知した箇所でそのインデックスエントリだけを削除する自己修復方式なら、追加の定期実行処理なしで矛盾が解消されていく

## インデックス設計

- 検索は**事前に定義したインデックスを経由してのみ**可能とする（生の行データを条件検索することはできない）
- 複合インデックス（複数カラムをまとめたインデックス）は、RDBMSのB-tree複合インデックスと同じ制約に従う。**先頭カラムのみ範囲検索が可能**で、それ以降のカラムは完全一致（等価）検索のみ対象となる
- インデックスの構造は「キー名の並び」と「値の並び」を分離する。
  - プレフィックス部分: `{key1}!{key2}!.../` … インデックス定義（どのカラムを、どの順で）を表す
  - エントリ名部分: `{value1}!{value2}!.../{行ファイル名}` … 実際の値（`!`区切り）＋行ファイル名への参照

## 値のエンコード方式

インデックスのエントリ名に使う値は、S3キーとして安全かつ、範囲検索が必要なカラム（先頭カラム）については辞書順ソートが数値順・時系列順と一致するようにエンコードする。

### 文字列

- 値のUTF-8バイナリを**そのままhex化する**（長さプレフィックス等は付与しない）。通常の文字列比較は「先頭バイトから順に比較し、差が出た時点で決まる」だけなので、hex化（1バイト＝2文字の固定変換）でも通常の辞書順比較と完全に一致し、`gt`/`gte`/`lt`/`lte`（範囲演算子）もそのまま使える。また短い文字列が長い文字列のprefixになっている場合に短い方が先、という関係もそのまま保たれる（例: `"ab" < "abc"`）
- インデックス対象の文字列はUTF-8で**255バイトまで**に制限する（S3キー長の上限(1024バイト)や他の構成要素との兼ね合い）。行データ自体（非インデックスカラム）には制限を設けない
- いずれにしても、LIKE検索・部分一致は引き続きできない（別途N-gramインデックス等の対応が必要、対象外）。短い文字列であれば実質的にこの制約は問題になりにくい

以下の案は、実際に検証した結果、当初検討していたが不採用にした。

- **標準`base64`でのエンコード**: 標準base64のアルファベットは文字コード順とビット値順が一致しておらず、辞書順を保持しないため範囲検索に使えない（llrtが`base64url`エンコーディング名を未サポートという事情もあり、そもそも手動のURL-safe変換が必要だった）
- **長さプレフィックス＋UTF-8バイナリのhex化**: 桁数の少ない文字列が内容によらず常に先に来てしまい、通常の辞書順と一致しなくなる（例: 通常は`"Ab" < "B"`だが、長さ優先だと`"B"`が先に来て逆転する）

### 数値（整数・浮動小数点）

- 固定バイト数（例: 64bit整数なら8バイト）にバイナリ変換してから、文字コード順と値の昇順が一致する固定長エンコード（`tools/xor128.js`の`_toString64`と同じ`_CODE64`アルファベット、もしくはhex）で表現する
- **先頭カラム（範囲検索対象）は固定長エンコードが必須**。後続カラムは`!`区切りによって値の混同（例:「1」+「23」と「12」+「3」が同じ文字列になる曖昧さ）は防げるため、厳密な固定長化は必須ではない
- **負の値の範囲検索にも対応する**（金額の増減など、マイナス値も範囲検索したいという要件があるため）。符号ビットを反転させてから固定長エンコードする2の補数ベースの補正を行う。

  以前検討した`Number.MAX_SAFE_INTEGER - (num + 1)`という変換方式は、実際に検証した結果、以下の2つの不具合があることが確認できたため**採用しない**。
  1. 64進数変換部分(`ret`)がゼロ埋め（固定長化）されておらず、同じ桁数バケット内でもソート順が崩れる（例: `10`が`99`より後にソートされる）
  2. `Number.MAX_SAFE_INTEGER`に値を加減算する変換が、JSの安全な整数範囲(2^53)を超えて精度が崩れ、異なる負数が同じ文字列にエンコードされてしまう（`-9`と`-10`が衝突）

  代わりに、`modules/csv/jsonb.js`が既に行っている「DataView/Bufferで数値を固定長バイナリ化する」実装を踏襲し、固定長バイナリの符号ビットをビット単位で反転させてから固定長エンコードする方式を採用する。

  具体的な符号ビット反転ルールは以下の通り（実際にNode.jsで動作検証済み。負数・0・Infinity・最小/最大値を含む代表値でエンコード後の文字列順が数値順と完全一致することを確認した）。

  - **整数**（2の補数表現）: 符号ビットのみを反転する（`bits ^ (1n << 63n)`）。2の補数はもともと符号ビット以外の並びが単調増加のため、符号ビットの反転だけで正しくソート可能になる
  - **浮動小数点**（IEEE754倍精度）: 符号ビットが0（非負）の場合は符号ビットのみ1に反転する。符号ビットが1（負）の場合は全ビットを反転する。非負のIEEE754浮動小数点は、ビットパターンをそのまま符号無し整数として比較すれば数値順と一致するという性質があるため、符号ビットを立てるだけで「負の変換後の値 < 非負の変換後の値」という並びを実現でき、負の値側は全ビット反転によって「絶対値が大きい負数ほど小さい変換値になる」という順序が得られる

  変換後の64bit値は、固定長バイナリとして扱い、`_CODE64`（もしくはhex）でエンコードする。

### 真偽値

- `t`(true) / `f`(false) の1文字で表現する

### NULL値

- カラムは基本的に`not null`（値が必ず存在する）を前提とする
- どうしてもNULLをインデックス対象にしたい場合は、`~null~`のような特殊な固定文字列を予約済みトークンとして使う。`~`はBase64（および手動でのURL-safe変換後の`A-Za-z0-9-_`）には出現しない文字なので、通常のエンコード結果と衝突しない
  - 数値の固定長エンコードは`_CODE64`（`~`を含むアルファベット）を使うため、理論上`~`自体は数値エンコード結果にも出現し得るが、数値エンコードは常に固定長（カラムの型ごとに長さが確定）であるため、`~null~`（固定6文字）とは長さが一致せず、実装上は「まず`~null~`との完全一致を確認してから、型ごとのデコード処理を行う」という順序にすれば曖昧さは生じない

## モジュール名・配置

- モジュール名は `s3IndexTable.js` とする
- 配置場所は、既存の`modules/s3table/s3MasterTable.js`と同じ系統（AWS-SDK-V3を使う`modules/sdk/`配下）に`modules/s3table/s3IndexTable.js`として追加する
- `table.json`（全テーブル分のテーブル定義・インデックス定義を集約したファイル）はJSON形式とし、テーブル名をキーに`createTable`に渡すオブジェクト形式をそのまま保存する

## クエリの制約

- 検索は事前定義済みのインデックス経由でのみ可能。インデックス化していない条件では検索できない
- 複合インデックスは先頭カラムのみ範囲検索可能。それ以降は完全一致のみ
- 上記の制約により、RDBMSほど自由な条件検索はできない。この点は「1行1ファイルによる書き込み競合の少なさ・低コスト」というメリットとのトレードオフとして許容する

## 大量件数を絞り込む場合の性能方針

インデックスで絞り込んだ候補件数が多い場合（例: 100万件）に、残りの条件（等価一致）をどう処理するかは以下の方針とする。

- **候補件数が少ない場合（数千件程度）**: 候補行を素直にGetObjectしてJSONパースし、アプリケーション側で残りの条件をフィルタしてよい。コスト（GET $0.0004/1000件）・時間（並列実行で数秒程度）ともに問題にならない
- **候補件数が多い場合（例: 100万件）**: 全件GetObjectする方式は非現実的（コスト・Lambdaの実行時間制限の両面で破綻する）。**残りの条件についても別途インデックスを用意し、LIST操作（本文転送が発生しない、メタデータのみの操作）だけで最終候補まで絞り込む**必要がある
- そのため、想定されるクエリパターンごとに、適切な複合インデックス（等価一致条件を先頭に、必要なら範囲検索条件を後続に）を設計しておくことが前提となる。これはDynamoDBなど他のNoSQL/KVS型データベースにも共通する制約である

## API設計

既存の`modules/s3table/s3MasterTable.js`・`modules/csv/memoryTable.js`のAPIスタイルを踏襲しつつ、本設計の制約（インデックス経由のみ検索・複合インデックスは先頭のみ範囲検索・updateは削除＋再作成）に合わせた形とする。

### テーブル・インデックスのライフサイクル

~~~js
// テーブル作成（カラム定義＋インデックス定義を同時に指定）
await db.createTable("users", {
  columns: {
    id: { type: "int", notNull: true },
    name: { type: "string", notNull: true },
    email: { type: "string", notNull: true },
    age: { type: "int", default: 0 },
    createdAt: { type: "date", default: () => new Date() },
    profile: { type: "json" }
  },
  indexes: {
    // インデックス名: [カラム名の並び]（先頭カラムだけが範囲検索対象）
    byId: ["id"],
    byEmail: ["email"],
    byAgeCreated: ["age", "createdAt"]
  }
});

await db.dropTable("users");
await db.createIndex("users", "byAge", ["age"]);
await db.dropIndex("users", "byAge");

// 追加API(テーブル管理コマンド bin/tableTool 向け).
await db.listTables();                 // 全テーブル分の定義を{テーブル名: schema}で取得
await db.alterColumns("users", {...}); // カラム定義を丸ごと差し替え(データは変更しない)

// バックアップ/リストア(テーブル管理コマンド bin/tableTool 向け).
const backup = await db.backupTable("users");   // { tableName, backupId, rowCount, indexEntryCount }
await db.listBackups("users");                  // 既存バックアップ世代(backupId)一覧(古い順)
await db.restoreTable("users", backup.backupId); // 指定世代の内容でテーブルを全置換
await db.previewRestore("users", backup.backupId); // restoreTableのdry-run(行数比較のみ、復元しない)
await db.pruneBackups("users", 5);                 // 直近5世代だけ残し、古い世代を削除
~~~

- `dropTable`は`table/`と`index/`配下の全オブジェクトを削除するため、行数分のDeleteObjectを要する
- 既存テーブルへの`createIndex`は、既存の全行に対してインデックスエントリを遡って作成するバックフィル処理を要する
- いずれも[想定スケール](#想定スケール)の通り、1万件程度までを前提とした許容範囲のコストとする
- `listTables`/`alterColumns`は、`bin/tableTool`(createTable/dropTable/alterTable/alterIndex)が現在のテーブル定義との差分を検出・適用するために使用する(詳細は`bin/README.md`のtableToolコマンド節を参照)
- `backupTable`/`listBackups`/`restoreTable`/`previewRestore`/`pruneBackups`は`s3MasterTable.js`と共通の機能(両モジュールとも同名APIを提供する)。行データ(`table/{tableName}/`)・インデックスエントリ(`index/{tableName}/`、`s3MasterTable.js`には無い)・スキーマ定義を`backup/{tableName}/{backupId}/`配下(`backupId`は実行時のUnixTimeミリ秒)にそのまま複製する物理コピー方式で、S3の`CopyObject`は使わず既存の`get`/`put`経由で複製する(コスト度外視・実装のシンプルさを優先)。複数世代を保持でき、`restoreTable`は指定世代の内容で現在のテーブルを**全置換**する(差分マージはしない。事前に現在の`table/`・`index/`配下を全削除してから復元する)。バックアップ/リストア実行中も通常のCRUD処理自体はロックされないため、整合性の取れたバックアップ/リストアを行うにはメンテナンス時間帯に実行する運用が前提となる(`bin/tableTool`のメンテナンスロックにより、他の管理コマンドとの多重実行のみ防止される)。詳細な試算は`docs/memo/s3IndexTable-cost-estimate.md`も参照(試算内容自体はs3IndexTable.js固有のI/Oコストが対象)
- `previewRestore`は`restoreTable`実行前のdry-run。現在の行数とバックアップの行数を比較するだけで、実際の復元・削除は一切行わない(行数以外のスキーマ差分等は表示しない)
- `pruneBackups`は`listBackups`で古い順に取得した世代のうち、直近`keep`世代を残して古いものを削除する。`keep`以下の世代数なら何もしない

| タイプ名 | 説明 | インデックス対応 |
|---|---|---|
| `string` | 文字列 | ○（base64＋URL-safe変換でエンコード） |
| `int` | 整数 | ○（固定長バイナリ＋符号ビット反転） |
| `float` | 浮動小数点 | ○（同上、IEEE754） |
| `boolean` | 真偽値 | ○（`t`/`f`） |
| `date` | 日付・日時。内部的には`int`（UnixTimeミリ秒）と同じ扱いだが、insert時にJSの`Date`オブジェクトを受け取り、select時も`Date`オブジェクトとして返す糖衣構文 | ○（`int`と同じエンコード） |
| `json` | 任意のJSON値（オブジェクト・配列など） | ×（インデックス非対応。行データとしてのみ保持） |
| `seqId` | Snowflake ID方式のユニークID（固定長16桁の小文字hex文字列）。insert時に値省略で自動生成される | ○（`string`と同じエンコード） |

インデックス対象カラムは、`json`型以外のいずれかを使う前提とする。

カラムのオプションは`notNull`（必須項目）と`default`（省略時のデフォルト値、関数も指定可）の2つのみサポートする。既存の`modules/s3table/s3MasterTable.js`が持つ`primaryKey`/`unique`/`autoIncrement`は、本設計では**提供しない**。

- `primaryKey`/`unique`の一意性制約は、挿入時に「既に同じ値が無いか」を確認してから書き込む必要があるが、複数リクエストが同時に走ると確認と書き込みの間に競合（TOCTOU）が起きるため保証できない
- `autoIncrement`（連番管理）はロック無しでは競合するため、この設計とは相性が悪い。行の一意性は「UnixTime+ナノ秒+LambdaID+乱数」による行ファイル名で担保する方針のままとする
  - 【追記】後日、連番的な採番が必要なユースケース向けに`type: "seqId"`（Snowflake ID方式。`modules/s3table/seqId.js`）を型システムに追加した。これはロック・中央採番管理を一切必要とせず各Lambda実行環境が独立して生成できるため、上記のTOCTOU競合の問題を回避できる。`string`型と同じエンコード方式（固定長16桁hex文字列のUTF-8バイナリhex化）でインデックス対応も可能。詳細は`docs/s3MasterTable.md`のseqId型の節を参照

### INSERT / SELECT / UPDATE / DELETE

`where`は「インデックス名をキーとし、値にそのインデックスの条件を持つオブジェクト」という形にする。`index`という別フィールドを持たず、`where`のキー自体がどのインデックスを使うかを表す。

~~~js
// INSERT
await db.insert("users", { id: 1, name: "Alice", email: "a@example.com", age: 30, createdAt: Date.now() });

// SELECT（先頭カラムのみ範囲演算子(gte/lte等)、以降は等価のみ）
const rows = await db.select("users", {
  where: {
    byAgeCreated: { age: { gte: 20, lte: 40 }, createdAt: 12345678 }
  },
  orderBy: { byAgeCreated: "asc" },   // 先頭カラムのみソート指定可能
  limit: 100
});

// 複数インデックスをまたぐAND条件も、whereに複数キーを並べるだけで表現できる.
// 実装上は各インデックスでLISTした候補行ファイル名の積集合を取る.
const rows2 = await db.select("users", {
  where: {
    byEmail: { email: "a@example.com" },
    byAge: { age: { gte: 20 } }
  }
});

// UPDATE（内部的には対象行を検索→削除→新規行として再作成）
await db.update("users", { where: { byId: { id: 1 } } }, { age: 31 });

// DELETE（対象行を検索→行ファイルを即時削除。インデックスは自己修復に任せる）
await db.delete("users", { where: { byId: { id: 1 } } });
~~~

- `where`が空、またはインデックス名に一致するキーが1つも無い場合は**エラーとする**。検索は必ず何らかのインデックス経由でなければならない、という制約がこの形式によって自然に強制される
- `orderBy`に指定したインデックス名が`where`に含まれていない場合（絞り込みには使わないが並び順だけ別インデックス基準にしたい場合）は、S3のLIST順をそのまま使えないため、取得後にアプリ側でメモリソートする（[想定スケール](#想定スケール)の前提であればコストは無視できる）
- `update`は行の物理ファイル名が変わる設計のため、その行が持つ全インデックスエントリを毎回作り直す必要がある（値が変わっていないインデックスも含む）。実質「変更カラムだけ更新」ではなく「行の削除＋全再作成」相当のコストになるが、[想定スケール](#想定スケール)の前提の範囲では許容する

### where の演算子仕様

`modules/csv/memoryTable.js`の演算子体系を参考にするが、そのまま全部を持ってくるのではなく、**S3のLIST操作に効率よく変換できるものだけ**に絞る。

先頭カラム（範囲検索可能）:

| 演算子 | 意味 | S3側の実装 |
|---|---|---|
| `eq` | 一致 | 完全一致prefixでLIST |
| `gt` / `gte` / `lt` / `lte` | 大小比較 | `StartAfter`/範囲prefixでLIST（`gte`+`lte`を同じ条件オブジェクトに書けば範囲検索になる） |
| `in` | 複数値のいずれか | 値ごとに個別LISTして結果をunion（値の数だけLIST回数が増える） |

後続カラム（等価一致のみ）:

| 演算子 | 意味 | S3側の実装 |
|---|---|---|
| `eq` | 一致 | プレフィックスの一部として直接組み込み |
| `in` | 複数値のいずれか | 値の数だけ個別LISTしてunion |

`memoryTable.js`から意図的に採用しないもの:

- **`ne`（不一致）**: 「値未満」と「値超え」の2つのLIST範囲に分割してunionする必要があり、コストの割に用途が少ないため見送り
- **`regexp`・文字列の部分一致**: S3のキー比較には落とし込めず、結局全候補をGetObjectして本文を見るしかなくなり、このデータベースのメリット（LISTだけで絞り込む）を失うため対象外。部分一致検索が必要な場合はN-gramインデックス等の別途対応が要るが、今回はスコープ外とする
- **文字列長比較(`leq`/`lgt`等)・Date/Timestamp専用演算子群**: 数値カラムとして統一的に`gte`/`lte`で扱えるため、別体系を用意する必要がない
- **`between`という専用演算子**: `{ gte: x, lte: y }`を同じ条件オブジェクトに書けば同じことができるため、別名を用意しない

### ページング（offset / limit）

`select`は`limit`とあわせて`offset`もサポートする。

~~~js
const rows = await db.select("users", {
  where: { byAgeCreated: { age: { gte: 20, lte: 40 } } },
  orderBy: { byAgeCreated: "asc" },
  offset: 50,
  limit: 50
});
~~~

S3の`ListObjectsV2`はカーソル（`ContinuationToken`）ベースのページングしかサポートしておらず、「先頭からN件スキップ」という機能は無い。そのため`offset`は、内部的に`ListObjectsV2`を複数回呼び出して該当件数まで読み飛ばす実装になる（1回のLISTで最大1000件しか進められない）。これによる既知の制約は以下の通り。

- **レイテンシ**: ContinuationTokenは前の呼び出し結果に依存するため、読み飛ばしは並列化できず直列実行になる。[想定スケール](#想定スケール)（最大1万件）であれば最悪でもLIST呼び出し10回程度で、後ろのページほど徐々にレイテンシが増える体感差はあるが致命的ではない
- **一貫性**: ページをまたいで閲覧している間に行の追加・削除が起きると、ソート順の途中に割り込む形になり、行の重複表示・欠落が起こりうる。厳密なスナップショット一貫性は保証しない
- **複数インデックスのAND条件との組み合わせ**: `where`で複数インデックスを指定している場合、offsetで読み飛ばす際も候補ごとに他インデックスとの合致確認が必要になるため、単一インデックスの場合より読み飛ばしコストが増える

これらはいずれも「補助的なページング機能」という位置づけの範囲内の制約として許容する。

### GROUP BY / 集計関数（SUM, COUNT, AVG, MIN, MAX）

S3側で集計はできないため、`memoryTable.js`と同様に「インデックスで絞り込んだ結果セットを取得してから、アプリ側（JS）で集計する」方式にする。

~~~js
const stats = await db.select("users", {
  where: {
    byAgeCreated: { age: { gte: 20, lte: 40 } }
  },
  groupBy: ["age"],
  aggregates: { count: { fn: "count" }, avgCreatedAt: { fn: "avg", col: "createdAt" } }
});
~~~

これは絞り込み後の候補件数が数千〜1万件程度であることを前提とした実装であり、[大量件数を絞り込む場合の性能方針](#大量件数を絞り込む場合の性能方針)で述べた通り、より大規模なデータには向かない。
