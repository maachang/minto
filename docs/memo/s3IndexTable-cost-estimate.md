# s3IndexTable.js のS3 I/Oコスト試算メモ

このメモは「s3IndexTable.js（1行=1ファイル方式）を使った場合、S3のリクエスト課金だけで
月額1,000円を超えるにはどれくらいのI/O量が必要か」を試算した結果を記録するもの。

前提条件:

- 環境: AWS Lambda（東京リージョン）から東京リージョンのS3を利用（同一リージョンのため
  データ転送料金は発生しない）
- 為替レート: 1ドル=160円換算
- 対象モジュール: `modules/s3table/s3IndexTable.js`
- テーブル規模: 1テーブルあたり1万行程度を想定
- インデックス数: 2本（典型例として仮定。実際のテーブル定義のインデックス数に応じて
  以下の「N」を読み替えること）

## 東京リージョンのS3実料金（AWS公式Price List APIで確認済み）

`https://pricing.us-east-1.amazonaws.com/offers/v1.0/aws/AmazonS3/current/ap-northeast-1/index.json`
を取得し、`usagetype`が`APN1-Requests-Tier1`（PUT/COPY/POST/LIST）・
`APN1-Requests-Tier2`（GET/SELECT等）・`APN1-TimedStorage-ByteHrs`（ストレージ）の
価格を実データから抽出して確認した（2026-07-17時点のスナップショット。価格は改定され
得るため、正確な最新値が必要な場合は都度確認すること）。

| 項目 | 単価 |
|---|---|
| PUT/COPY/POST/LIST（Tier1） | $0.0047 / 1,000リクエスト = $0.0000047/回 |
| GET/SELECT等（Tier2） | $0.0037 / 10,000リクエスト = $0.00000037/回 |
| DELETE | 無料（価格表にSKU自体が存在しない） |
| ストレージ（Standard、先頭50TBまで） | $0.025 / GB・月 |

閾値: 1,000円 ÷ 160円 = **$6.25/月**

## s3IndexTable.jsの操作あたりS3リクエスト数（実装ベース）

1行=1ファイル＋インデックス=0バイトファイルという設計のため、インデックス数`N`が
コストに直結する。

| 操作 | S3リクエスト内訳 |
|---|---|
| `insert` | (1+N)回のPUT（行データ1回＋インデックス分N回） |
| `select`（1件のeq検索） | 1 LIST（インデックスの絞込） + 1 GET（行データ） |
| `select`（全件相当、1万行） | 約10 LIST（1000件/ページ、ListObjectsV2のページング） + 10,000 GET |
| `update`（1行） | 1 LIST（絞込） + 1 GET（旧行） + (1+N) PUT（新行＋新インデックス） + DELETE（旧行、無料） |
| `delete`（1行） | 1 LIST（絞込） + DELETE（無料） |

ストレージは1万行（1行数百byte想定なら数MB程度）なら月額1円未満で、今回の閾値には
事実上関係ない（閾値到達には約250GBの保存が必要）。

## 月1,000円を超えるのに必要な回数の目安（インデックス2本の場合）

- **insert中心**: 1件あたり3 PUT×$0.0000047=$0.0000141 → 約44万件/月
  （1日あたり約1.5万件、1時間あたり約615件）
- **select(1件検索)中心**: 1件あたり$0.00000507 → 約123万回/月
  （1日あたり約4.1万回）
- **select(全件スキャン相当)中心**: 1回あたり約$0.003747 → 約1,670回/月
  （1日あたり約56回、1時間に2〜3回程度）← 最も高コストな操作
- **update中心**: 1件あたり約$0.0000192 → 約33万件/月
  （1日あたり約1.1万件）
- **delete中心**: 実質LISTコストのみ → 約133万件/月
  （DELETE自体は無料なので絞込のLIST回数がボトルネック）

## 注意点

- 全件スキャン的なselect（orderByがwhereに含まれるインデックス以外を使う場合や広い
  範囲検索）は、LISTのページ数＋GET件数が跳ね上がるため、他の操作より圧倒的に少ない
  実行回数でコストが積み上がる。実運用でのアクセスパターン次第で閾値到達までの回数は
  大きく変わる。
- これはS3のI/Oコストのみの試算。実際にはLambdaの実行時間・呼び出し回数の課金も
  別途乗ってくるため、月額の実コストはLambda側が支配的になるケースが多いと考えられる
  （この試算にはLambda分を含めていない）。
- 上記は「インデックス2本」という仮定での試算。実際のテーブル定義のインデックス数に
  応じて`insert`/`update`のPUT回数（1+N）を読み替えること。
- `select()`の`groupBy`/`aggregates`（`count`含む）は、`_resolveCandidates`（LIST）の
  時点で該当行数（`candidateSet.size`）が分かっているにも関わらず、現状の実装では
  集計前に該当行を`_getRow`で全件GETしてから件数を数えている。そのため`count(*)`相当の
  クエリだけを投げても「全件スキャン相当」と同じコスト（GET分含む）がかかる。GETを
  省いて`candidateSet.size`をそのまま返すショートカットを追加すれば安くなるが、
  2026-07-17時点では未対応（改修する場合は事前に方針確認のうえ着手すること）。

## まとめ（DynamoDB等との比較・S3IndexTableの利点）

- **DynamoDBとの比較**: DynamoDBもオンデマンドで書き込み/読み込みユニット単位の課金＋
  ストレージ別建てだが、s3IndexTableはリクエスト単価自体が非常に安く
  （PUT $0.0000047/回、GET $0.00000037/回）、素朴なCRUD中心の使い方であれば
  DynamoDBより割安に収まりやすいと考えられる。
- **S3の耐久性**: S3自体の可用性・耐久性（11 9s）はDynamoDBと同等以上であり、
  自前でRDBMSサーバーを運用する場合のディスク故障・バックアップ運用の心配が
  要らない点は大きな利点。1行=1ファイルという設計も、部分的な破損が他行に
  波及しないという意味で相性が良い。
- **総評**: 「使えば使うほど金がかかる」が「普通に使っている分にはそんなに
  お金はかからない」従量課金モデル。ただし全件スキャン系の操作（count含む）だけは
  他の操作に比べて桁違いにコストが伸びやすいため、そこだけ注意して設計・利用すれば
  十分実用に足るコスト感、という結論。
